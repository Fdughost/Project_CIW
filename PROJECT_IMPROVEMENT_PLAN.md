# Chloe in Wonderland 项目改进方案

## 总体方向

把当前线性网页解谜升级为“每章一个可操作的故事道具/小游戏”的互动叙事体验。保留温柔童话、生日礼物、Chloe 与 Alex 的主线，让玩家通过观察、点击、拖拽、排序、试错和揭示来推进，而不是只在通用表单里输入答案。

默认节奏：

1. 章节场景与故事铺垫。
2. 出现一个核心道具或小游戏。
3. 玩家与道具互动解谜。
4. 成功后触发章节事件反馈。
5. 道具或事件自然引出下一章。

提示采用逐级展开：轻提示、结构提示、接近答案提示。默认不直接展示提示。

## 全局玩法系统

- 每章统一结构为：章节场景 -> 关键道具 -> 解谜交互 -> 成功反馈 -> 下一章转场。
- 每个谜题支持错误反馈、尝试次数、重置当前谜题、键盘/触控双端操作。
- 进度条后续应不只显示第几关，还应显示已获得线索物：金币、利是、手机、挂饰、借阅卡、卡片等。
- 成功反馈用剧情事件表达，例如门锁打开、守卫转身、电脑运行成功、手机解锁、挂饰合拢。
- 难度曲线：第 1-2 章轻推理，第 3-6 章互动性加强，第 7-8 章回到情感与文本解谜，最后温柔收束。

## 第一章：空之轨迹

核心目标：把“看乐谱输入四位密码”变成“在小木屋里发现口琴并推导门锁密码”。

玩法方案：

- 场景左侧是小木屋叙事，右侧是木门、四位机械密码锁、桌上的口琴和乐谱。
- 第一章背景音乐使用 `寻道树 - 星之所在（口琴版）（Cover う～み）`，只在第一章播放，离章淡出。
- 玩家点击口琴后，乐谱从半透明状态变清晰；点击或播放乐谱时音符逐个高亮。
- 密码锁做成四个滚轮，每个滚轮可以上下切换数字，比文本输入更像真正开锁。
- 乐谱旁放“试吹一遍”按钮，模拟音符高亮节奏。
- 解谜路径：玩家从简谱中找出关键音符，推导四位数字，再拨动门锁。
- 成功反馈：门锁齿轮转动，壁炉火光变亮，木门打开，室外光线涌入，音乐淡出。

提示设计：

- 第一提示：口琴不是装饰。
- 第二提示：留意被火光描深的音。
- 第三提示：答案来自四个被强调的音符。

验收标准：

- 第一章答案以当前代码和乐谱线索为准：`7564`。
- 策划文档中的 `5564` 是旧稿错误，不再采用。
- 不看源码也能通过观察乐谱推导答案。
- 玩家可以只用鼠标/触控完成，不必键盘输入。
- 答错不会清空进度，只让门锁轻微反馈并给温和提示。

## 第二章：谁是骑士？

核心目标：把 A-G 选项按钮升级为“守卫证词推理桌”。

当前旧文案逻辑不成立：在“只有一个骑士说真话，其余说谎”的规则下，旧证词无法推出唯一答案。应替换为以下已校验版本，唯一解仍为 `G`：

```text
A： "黄金在 A 或 B 的身后。"
B： "黄金在 C 或 D 的身后。"
C： "黄金在 E 或 F 的身后。"
D： "黄金不在 G 的身后。"
E： "黄金在 B 或 E 的身后。"
F： "黄金在 A 或 C 的身后。"
G： "黄金就在我的身后。"
```

校验结论：

- 若黄金在 `G` 身后，只有 `G` 的话为真。
- 若黄金在 A-F 任一处，真话数量都大于 1。
- 因此在“只有一个骑士”的前提下，唯一答案是 `G`。

玩法方案：

- 画面展示 7 张守卫卡片，每张有角色剪影、字母、证词。
- 玩家可以给每张卡打标记：真话、谎话、待定。
- 旁边提供推理板，显示当前假设下是否恰好只有一句真话。
- 最终玩家把骑士徽章交给某位守卫，或点击守卫后确认。
- 错误选择不只显示“错误”，而是提示“这个假设下真话数量不符合规则”。
- 成功反馈：G 守卫转身，背后露出黄金硬币和新年利是；两个物品飞入进度背包，为第三章铺垫。

提示设计：

- 第一提示：只有一句话能为真。
- 第二提示：检查“黄金在 G 身后”的假设。
- 第三提示：如果 G 为真，A-F 的区间判断都不成立。

验收标准：

- A-G 每个证词在界面上可见，不需要玩家回看大段故事。
- 玩家必须做出明确选择才能过关。
- 答案 G 的推理路径在成功反馈里能被简短解释。

## 第三章：码农天堂

核心目标：把占位密码 `0222` 替换成“修复码农工作流”的小游戏。

玩法方案：

- 电脑屏幕里出现小型工作流看板：需求、设计、编码、测试、发布、生日礼物。
- 初始状态是混乱的：任务顺序错误、测试节点错位、发布节点被堵住。
- 玩家通过拖拽任务卡，或用上下按钮排序，让礼物能从需求一路流到发布。
- 推荐答案机制：流程跑通后自动过关，不再输入 `0222`。
- 失败反馈：点击“运行流程”后，错误节点闪烁，主管给出一句具体阻塞原因。
- 成功反馈：电脑里出现生日礼物构建成功动画，主管扔出砖头手机，引出第四章。

提示设计：

- 第一提示：礼物要先有需求再写代码。
- 第二提示：测试应该在发布前。
- 第三提示：按现实软件流程排序。

验收标准：

- 玩法和“码农天堂”主题强相关。
- 玩家能通过拖拽或点击完成，失败原因可视化。
- 第四章获得手机的剧情转场自然。

## 第四章：砖头

核心目标：把普通密码输入改成诺基亚手机解锁体验。

玩法方案：

- 显示一台可交互 Nokia 手机：屏幕、九宫格键盘、确认键 `#`、清除键。
- 玩家可以点击手机背面或翻转按钮，看到刻着 `LOVE`。
- 玩家可以打开长途国家编号列表：USA +872、JPN +576、CHN +246、HKG +454。
- 解谜路径：由国家缩写和区号推导手机键盘字母数字关系，再把 LOVE 转成 `5683`。
- 输入方式：玩家按下 5-6-8-3，再按 `#`。
- 错误反馈：屏幕显示“密码错误”，按键灯闪一下；不使用浏览器 alert。
- 成功反馈：手机屏幕亮起，出现“1 个未接来电”，点击回拨进入第五章。

提示设计：

- 第一提示：背面的 LOVE 很重要。
- 第二提示：国家缩写和区号不是随机的。
- 第三提示：把字母映射到手机键盘数字。

验收标准：

- 不出现通用文本框。
- 玩家可以从界面线索推导 `5683`。
- 电话回拨是过关后的下一步交互，而不是纯文本叙述。

## 第五章：尾行

核心目标：把“追上远处女生”做成轻量追逐/路径判断小游戏。

玩法方案：

- 场景是迷宫小径，玩家站在底部，女生身影在远处移动。
- 每一轮给出 2-3 个方向选择：左、前、右。
- 正确方向来自脚印、断枝、电话喘息声方向、飘落物等线索。
- 推荐做 5 步追踪序列，错一步不会立刻失败，而是拉开距离；错 2-3 次女生消失并重试当前章。
- 可加入轻微倒计时制造追逐感，但不做高压动作挑战。
- 视觉上用小地图或路径卡片表现，不必做复杂实时游戏。
- 成功反馈：女生身影停顿，脚下掉出银色挂饰，玩家拾起进入第六章。

提示设计：

- 第一提示：不要只看人影，看她留下的痕迹。
- 第二提示：最新的脚印颜色更深。
- 第三提示：高亮当前轮关键线索。

验收标准：

- 玩家每一步都能解释为什么选这个方向。
- 失败后重试成本低，不重放长故事。
- 结尾自然产出银色挂饰。

## 第六章：双剑合璧

核心目标：把银色挂饰做成旋转、拼合、对称识别谜题。

玩法方案：

- 屏幕中央是破碎挂饰，由两片心形或剑形金属片组成。
- 玩家可以拖动、旋转、翻面两片挂饰，让它们合成完整图案。
- 每片挂饰上有细微刻痕，合拢后刻痕组成符号或短句，触发第七章借阅卡。
- 操作方式：点击旋转 90 度，拖拽移动；靠近正确位置时有磁吸效果。
- 额外解谜点：两片不是简单拼图，必须根据“双剑合璧”的标题让两把剑形成心形。
- 错误反馈：合不上时两片轻轻弹开；Alex 提醒“方向对了，但好像还差一个角度”。
- 成功反馈：挂饰合拢发出清脆声，天上飘落借阅卡，进入第七章。

提示设计：

- 第一提示：它可能不只是挂饰。
- 第二提示：试着让两把剑形成一个心。
- 第三提示：显示半透明目标轮廓。

验收标准：

- 玩法不依赖输入密码。
- 拖拽在桌面和移动端都可用，移动端可用按钮旋转辅助。
- 成功动画明确连接到借阅卡。

## 第七章：借阅卡

核心目标：把填空题升级成“旧图书馆档案检索”。

玩法方案：

- 显示一张泛黄借阅卡，空缺处可点击放大。
- 旁边有一个小型卡片目录柜。
- 玩家根据 `年份：200_`、`类别：反差`、`主题词：将军`、`名字：我用 _ _ 报答爱` 找到答案。
- 交互一：年份数字用拨轮选择 `0-9`，正确为 `8`。
- 交互二：名字空缺用两个字填入，正确为 `所有`。
- 可选增强：目录柜里有几张候选作品卡，玩家通过筛选年份/主题词，最终翻到目标卡。
- 错误反馈：卡片上的字迹短暂变模糊，提示“年份和题名需要同时成立”。
- 成功反馈：借阅卡补全后，卡面浮现“所有”，Alex 接上第八章的 18 年回应。

提示设计：

- 第一提示：年份缺的是最后一位。
- 第二提示：主题词“将军”指向作品线索。
- 第三提示：答案是 2008 和“所有”。

验收标准：

- 两个答案必须按顺序验证，不能用数组乱序通过。
- 卡面视觉像真实旧借阅卡，而不是普通表单。
- 解出后强化“所有”这个情感关键词。

## 第八章：莫奈的画

核心目标：把最后填空变成“卡片显影/翻看”的情感揭示。

玩法方案：

- Alex 递来的卡片显示两行诗句：`余 _ _ _`、`余生 _ _ _`。
- 玩家可以拖动卡片、改变光照角度，或点击“放到月光下”，让隐藏字迹逐步显现。
- 第一层显现“余光”，第二层显现“是你”，第三层显现“也是你”。
- 最终组合成 `余光是你`、`余生也是你`。
- 在玩家填写前，卡片背面可出现“别看了，莫奈没说过这句话”，作为幽默反转。
- 输入可以保留，但应包装成卡片上的两个空行，而不是通用输入框。
- 成功反馈：卡片完整显影，背景从仙境回到生日祝福，兑换券以礼物形式出现。
- 错误反馈：不要生硬说错；可以说“这句话好像还没完整显影”。

提示设计：

- 第一提示：不是名人名言。
- 第二提示：第一句从“余光”开始。
- 第三提示：第二句回应“余生”。

验收标准：

- 最后一章情绪是温柔揭示，不是考试。
- 玩家完成前不要直接看到完整答案。
- 完成后兑换券出现要有仪式感，但动画克制。

## 实现接口建议

- `puzzleType` 扩展为：`harmonicaLock`、`guardLogic`、`workflowRepair`、`nokiaKeypad`、`chasePath`、`pendantMerge`、`libraryCard`、`cardReveal`。
- 每个 puzzle type 拥有自己的数据字段，例如选项、步骤、线索、正确序列、提示梯度、成功文案。
- 答案验证从 `Game.tsx` 抽到独立 helper，支持顺序、别名、标准化、章节自定义反馈。
- 每个谜题拆成独立组件，`PuzzleComponent` 只负责按类型分发。
- 章节数据继续作为单一内容入口，但复杂交互逻辑不要塞进 `gameData.ts`。

## 测试与验收

- 每章至少有一条 happy path 自动测试，确认能从第一章走到结尾。
- 答案验证单测覆盖：第一章数字、第二章选择、第四章 keypad、第五章路径序列、第七/八章有序填空。
- 移动端重点检查：挂饰拖拽、Nokia 按键、追逐方向选择、卡片显影。
- 视觉验收标准：每章看起来像不同道具，而不是同一个白卡片表单换文案。

## 默认假设

- 保留原 8 章故事主线和最终礼物揭示。
- 第 3、5、6 章按策划文档里的“电子游戏另外制作”补成网页内轻量小游戏。
- 第一章答案固定为 `7564`；策划文档 `5564` 是旧稿错误。
- 第二章使用新版已校验证词，答案为 `G`。
- 优先做轻量、稳定、浪漫的交互，不做高难度实时动作游戏。
